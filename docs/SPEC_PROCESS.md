# SPEC & PLAN 协作过程记录

## 概述

本文档记录了与 Superpowers brainstorming + writing-plans 技能协作，从零生成 `ws`（多仓库工作区管理器）完整 SPEC 与 PLAN 的过程。时间跨度：2026-05-31，单次对话内完成。

---

## 1. Brainstorming 关键节点

### 1.1 智能体追问的问题列表

| # | 问题 | 类型 | 用户回答 | 影响 |
|---|------|------|----------|------|
| 1 | 如何知道要管理哪些仓库？（配置文件/自动发现/CLI参数/混合） | 多选 | D（多工作区） | 直接决定了 `~/.ws/workspaces/` 目录结构和 `ws switch` 命令 |
| 2 | 微服务技术栈？（Node/多语言/Go-Java/容器化） | 多选 | D（容器化） | 确认了 build/health 不需要特定语言运行时，降低了 v1 复杂度 |
| 3 | sync 操作具体指什么？（纯Git/分支管理/完整同步/manifest） | 多选 | B，A 作为子集 | 明确了 v1 scope：fetch+merge+checkout+dirty检测 |
| 4 | build 功能到什么程度？（Docker构建/智能构建/依赖检查/极简封装） | 多选 | D（极简） | **关键决策**：工具不内置构建逻辑，只执行用户配置的命令 |
| 5 | health check 覆盖什么？（HTTP探测/容器状态/自定义命令/组合） | 多选 | C（自定义命令） | 与 build 一致的设计哲学：工具做编排，用户定义"做什么" |
| 6 | CLI 用什么语言？（Go/TypeScript/Rust/Python） | 多选 | A（Go + Cobra + Viper + Bubble Tea） | 锁定了所有技术选型 |
| 7 | 配置文件结构？（单文件/分层/极简/声明式） | 多选 | A（单文件），并主动给出了完整的 yaml 结构设计 | **关键贡献**：用户在此处一次性给出了 groups、per-repo 属性、全局配置、V2 字段的完整设计 |
| 8 | 仓库失败时的处理？（继续/即停/按组/可配置） | 多选 | D（可配置），并主动给出了 CLI 层、执行引擎层、TUI 层、退出码的完整规约 | **关键贡献**：用户在此处定义了 fail-fast 传播时序、信号量模型、cancel 行为 |
| 9 | CLI 命令顶层结构？（扁平/分组/操作优先/类似docker） | 多选 | C（操作优先），并主动给出了 11 个子命令、通用 flag、worktree 切换、完整示例 | **关键贡献**：用户一次性锁定了整个 CLI 设计 |

### 1.2 问题设计分析

**做得好的地方：**
- 问题由浅入深：先问"是什么"（仓库布局 → 技术栈 → 操作语义 → 细节设计）
- 多选降低了回答成本，同时给出了框架让用户思考
- 问题 7-9 触发了用户主动输出详细设计，说明问题设置成功引导了深入思考

**可改进的地方：**
- 问题 2（技术栈）和问题 6（语言选择）有部分重叠——如果先问语言再问技术栈会更自然
- 问题 4（build）和问题 5（health）可以合并为一个问题，因为它们有相同的设计哲学（极简 vs 内置）

## 2. 三轮关键迭代

### 2.1 第一轮：架构方案选择

**AI 提议：** 三个架构方案
- 方案 A：核心执行引擎 + 薄 CLI 层（推荐）
- 方案 B：独立操作模块（零耦合但重复）
- 方案 C：管道/工作流引擎（过度设计）

**用户决策：** 选择方案 A

**理由：** 核心引擎复用是解决痛点（减少重复操作）的关键——Executor 共享后，fail-fast、concurrency、timeout 只需实现一次。

**影响：** 锁定了 `internal/executor` → `internal/runner` → `internal/cli` 三层架构，以及 `internal/tui` 作为独立层。

### 2.2 第二轮：SPEC 设计逐节审查

**过程：** AI 按五节逐步呈现设计，每节等待用户确认。

| 节 | 内容 | AI 提议 | 用户修正/补充 |
|-----|------|---------|---------------|
| 1 | 项目结构 + 核心接口 | 目录树 + Executor/Runner/Task/Result 接口 | 全部接受 |
| 2 | workspace.yaml + 执行引擎 | YAML 示例 + Executor 伪代码 | **6 项补充**：default_branch、per-command timeout、workspace_name、V2 字段标注、确认并发模型正确、确认路径支持相对/绝对 |
| 3 | CLI 命令设计 | 命令层级 + flag 表 + sync 专属 flag | **2 项补充**：ws init 支持 `--name` 一键模式、工作区存储结构 `~/.ws/` |
| 4 | TUI 设计 + 退出码 | 布局 ASCII + 交互键 + 状态图标 + 退出码表 | **3 项补充**：skipped/cancelled 对退出码的影响规则、plain 模式补充 cancelled/skipped/warning 示例、Detail 列截断规则 |
| 5 | 错误处理 + 测试策略 | 6 种错误类型 + 4 层测试 + v1 边界 | **3 项修正**：sync v1 范围精简措辞、包目录树与测试路径一致性确认、TUI 测试描述微调 |

**关键模式：** 用户在每节都会给出"确认通过 + N 点补充/微调"的反馈。这种模式很高效——不需要推翻重来，只需要精确修正。说明 AI 的设计方向正确，细节上用户有更具体的想法。

### 2.3 第三轮：SPEC 终稿自审与修正

**AI 自审发现的问题：**
1. **build 规约矛盾**：SPEC 中写了"按 type 使用默认命令"，但用户在第 4 个问题中明确选择了"极简封装，不内置构建逻辑"。修正：移除默认命令逻辑，`type` 标注为 v1 仅标识用途。
2. 无 TBD/TODO，无占位符，内部一致。

**用户审查：** 直接批准进入 PLAN 阶段，无额外修正。

**AI 自审发现的问题：**
1. **Task 类型不一致**：`Task` 结构体最初定义为 `Repo Repo`（嵌套整个 Repo），但 executor 和 runner 代码中使用了 `Task.RepoName` 作为直接字段。修正：Task 改为 `RepoName string` + `Group string`，不嵌套完整 Repo。
2. 修复后类型链一致：T2（类型定义）→ T4（executor 使用 Task.RepoName）→ T5/T6（runner 设置 RepoName）→ T8-T11（CLI 传递 Task）。

**用户审查：** 要求将 PLAN 移至 `docs/PLAN.md`，并要求增加依赖标注和并行策略。

### 3.2 被用户推翻或修正的 AI 提议

| AI 提议 | 用户修正 | 原因 |
|---------|----------|------|
| build 按 `type` 使用默认命令（go→`go build`, node→`npm run build`...） | v1 不内置任何默认命令 | 用户在第 4 个问题中选择了"极简封装"，与 type 默认命令矛盾 |
| SPEC 路径 `docs/superpowers/specs/YYYY-MM-DD-...` | `docs/SPEC.md` | 用户偏好简洁路径 |
| PLAN 路径 `docs/superpowers/plans/YYYY-MM-DD-...` | `docs/PLAN.md` | 同上 |
| SPEC 10 个章节的标准模板 | 用户要求加入问题陈述、用户故事、验收标准、风险等 15 章 | 用户需要更完整的文档结构 |

| Visual companion 跳过 | 用户未反对（CLI 工具，确实不需要） | 判断正确 |

### 3.3 用户主动贡献的完整设计（非 AI 提议）

这些是用户在设计讨论中直接给出的详细规约，AI 的角色是记录和整合：

1. **workspace.yaml 完整结构**（groups、per-repo 属性、V2 标记、全局配置）
2. **Fail-fast 三层设计**（CLI flag → executor cancel → TUI 展示 + 退出码）
3. **CLI 命令体系**（11 个子命令、目标范围 flag、互斥规则、示例调用）
4. **退出码详细规则**（skipped/cancelled 不计入失败、monitor 特殊处理）
5. **Sync --checkout/--force 行为**（dirty 检测、stash 策略）

## 4. 反思：Superpowers Brainstorming 技能评估

### 4.1 做得好的地方

1. **一次一个问题** — 避免了"选项轰炸"（9 个问题逐个展开），每次用户只需思考一个维度，回答质量高
2. **多选降低门槛** — 大部分问题是 A/B/C/D 选择，用户不需要从零构思，选择后自然会补充细节
3. **设计确认节奏** — 五节逐节审查的模式让用户可以在每个局部做出微调，避免了"全部写完再推翻"的浪费
4. **自审发现矛盾** — AI 在写 SPEC 终稿时主动发现了 build 默认命令与用户早期选择的矛盾并修正。这个机制很关键——如果没有自审，这个 bug 会进入 PLAN
5. **完整文档输出** — 15 章 SPEC + 15 步 PLAN 的产出质量超出了简单问答能达到的水平
6. **用户占主导** — 技能正确地让用户在设计决策中处于主导地位，AI 做结构化呈现和查漏补缺

### 4.2 做得不好的地方 / 不满之处

1. **过程冗长** — 从开始到 SPEC 完成经过了 9 个问题 + 5 节审查 + 自审 + 终稿，对于一个有经验的开发者来说节奏偏慢。如果用户一开始就有清晰的想法（如本次），可以加速跳过部分引导性问题。
2. **缺少"跳过"机制** — 技能没有提供"我已经知道我要什么，直接进入设计"的快进路径。用户在第 7 个问题后开始主动输出完整设计，说明已经不需要多选了，但 AI 仍然继续按流程追问。
3. **Visual companion 尴尬** — 技能要求为每个可能涉及视觉的问题判断是否启用 visual companion。对于纯 CLI 工具来说这个判断是多余的。技能应该在第一步就判断"这是否是 UI 项目"来决定是否需要 visual companion。
4. **SPEC 路径默认不合适** — 技能默认的 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` 路径被用户两次拒绝（SPEC 和 PLAN 都是），说明默认路径对于小型项目过于冗长。可以提供简单路径和复杂路径两种选项。
5. **PLAN 颗粒度默认偏大** — 初始 PLAN 是 10 个粗粒度 task，用户要求更细粒度 + 依赖标注 + 并行策略。writing-plans 技能应该有明确的颗粒度指南（每个 task 是否可由一个 subagent 在一次会话内完成？），当前缺少这个检查。
6. **没有显式的"回退"提示** — 当用户在回答中给出了远超问题范围的完整设计时，AI 应该识别到并说"看起来你已经有了清晰的设计，我们可以跳过剩余问题直接进入设计阶段"。当前缺少这个智能判断。

### 4.3 项目适配性总结

| 维度 | 评分 | 说明 |
|------|------|------|
| 问题质量 | ★★★★☆ | 问题逐步深入，多选降低门槛 |
| 节奏控制 | ★★★☆☆ | 前期偏慢，缺少加速路径 |
| 设计转化 | ★★★★★ | 用户输入→结构化 SPEC 转化质量高 |
| 自审能力 | ★★★★☆ | 发现并修正了 1 处矛盾 |
| 灵活性 | ★★★☆☆ | 对用户"快进"需求的响应不够敏捷 |
| 文档产出 | ★★★★★ | 15 章 SPEC + 15 步细粒度 PLAN 质量高 |
| 整体效率 | ★★★★☆ | 单次对话内完成 spec→plan→准备执行，效率可接受 |

## 5. 关键决策时间线

```
T+0min   用户提出想法："多仓库工作区管理器 CLI"
T+5min   智能体探索项目上下文（空目录）
T+10min  Q1-Q3：仓库布局 → 技术栈 → sync 语义
T+20min  Q4-Q6：build → health → 语言选择
T+30min  Q7-Q9：配置文件 → 容错策略 → CLI 结构
T+45min  方案选择（A/B/C） → 用户确认方案 A
T+60min  SPEC 第 1 节（项目结构+接口） → 用户确认
T+75min  SPEC 第 2 节（workspace.yaml+执行引擎） → 用户 6 点补充
T+90min  SPEC 第 3 节（CLI 设计） → 用户 2 点补充
T+105min SPEC 第 4 节（TUI+退出码） → 用户 3 点补充
T+120min SPEC 第 5 节（错误处理+测试） → 用户 2 点修正
T+130min AI 自审 → 修复 build 默认命令矛盾
T+140min SPEC 终稿 → 用户批准
T+150min AI 写 PLAN → 用户要求调整路径、颗粒度、依赖标注
T+160min PLAN 重写 + 自审 → 用户批准 → 准备执行
```

**总耗时**：约 160 分钟（单次对话），产出了可执行的 SPEC + PLAN。
