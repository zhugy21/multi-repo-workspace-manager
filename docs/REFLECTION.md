**项目：** ws — 多仓库工作区管理器
**日期：** 2026-06-01

## 1. Superpowers哪些技能发挥了最大作用，哪些感觉是“形式大于实质”

**最有用的是Brainstorming。** 它没有让我一开始就写代码，而是先用9个问题把想法拉到一个具体的范围里。比如它问“build你想做到什么程度”，我选了“极简封装”，这个决定是在写任何代码之前做出来的——如果自己闷头写，多半会忍不住往里面加东西，最后超出v1范围。所以这个环节确实有效。

**Writing Plans也很有用，但有一个前提：颗粒度必须细。** AI最初给我的PLAN只有10个task，每个都太粗，一个task里塞了8个实现点，根本没法用。我改成了15个task、每个都有精确的文件路径和验证步骤，才变得可执行。这说明“有用”的前提是人类去修正它的输出。

**形式大于实质的：Code Review的某些评分项。** 比如它检查了“测试是否覆盖了所有分支”，但实际检查出来的问题（发现Duration的JSON tag写的是纳秒值但字段名写的是ms，导致数据损坏）确实是有价值的。但它也有很多检查是泛泛的，比如“有没有未使用的变量”——这种交给编译器就够了。总的来说，有用，但不能依赖。

## 2. TDD强制在AI协作下是阻碍还是放大器

**取决于规约是否清晰。** 当规约清晰的时候，TDD是放大器——因为测试用例写好了，AI只需要实现它，方向很明确。但规约模糊的时候，TDD就是阻碍——AI连该测什么都搞不清楚，先写测试这个要求就卡住了它。

我的T2卡死就是一个典型例子：AI面对“实现pkg/types共享类型”这种描述时，根本不知道该写什么类型，也就无法写出第一个测试。一卡就是30分钟。我改成在PLAN里直接写出精确的Go结构体代码后，同一个AI在30秒内就完成了全部验证。

**所以TDD本身没问题，问题是规约的质量决定了TDD是否生效。** 如果你指望AI自己去理解和分解任务后再做TDD，目前还不太现实。

## 3. Subagent能自主跑多久？什么样的task拆解最优

**数据点：** 我派发了14次Subagent，其中有1次卡死超过30分钟（T2），其他都能在10分钟左右完成。

**卡死的那个task特点：** 描述抽象、没有给代码样例、任务里包含了多个实现点（8个点塞在一个task里）。
**最优的那些task特点：** 每个task写清楚了要修改的具体文件、代码大概长什么样、验证步骤具体是什么。

我的结论是：**一个task应该能在单次Subagent会话内（约5–10分钟）完成，包含不超过2–3个文件。** 超过这个范围就容易出问题。

## 4. SPEC/PLAN质量如何影响实现？有没有跑偏案例

**有，T2就是最典型的跑偏案例。** 当时PLAN里T2的描述是“实现struct with YAML+JSON tags”——这个描述对人类来说可能够了，但对AI来说远远不够。它花了很多时间在“猜测”上，最后猜不出来就卡死了。

**而我把它改成下面这样之后：**

// Repo 结构体
type Repo struct {
Name    string yaml:"name" json:"name"
Path    string yaml:"path" json:"path"
URL     string yaml:"url,omitempty" json:"url,omitempty"
// ...
}

AI直接复制进去，编译通过，测试全过。这个对比很直接地说明了问题：**PLAN的质量决定了Subagent的产出质量，而“对AI友好的PLAN”必须比我们平时写文档要更具体。**

## 5. 最有效的Prompt/Context策略

**就是“精确代码规格”策略。** 在PLAN里直接给出实现代码，让Subagent从“设计+写代码”降级为“复制+验证”。这个听起来像是人类干了很多活，但实际上它节约了更多时间——T2从30分钟卡死变成了30秒完成。

另一个有效的小策略：**在每个task的开头注入环境变量。** 比如`export PATH=/usr/local/go/bin:$PATH`——因为我看过一次AI因为找不到go命令而卡住，之后就在所有任务描述前面都加上了这一行。

## 6. 是否使用了Open Design

没有。我的项目是纯CLI工具，豁免了这个要求。

## 7. 如果重做，会改变哪些做法

1. **更早做冷启动验证。** 我是在写完完整SPEC+PLAN之后才做的。如果重做，我会在写完核心模块（比如Executor引擎）的SPEC后马上做一次微冷启动，早点发现规约里的歧义。

2. **减少对“自主推进”的期待。** 刚开始我会让Subagent自己去理解task并“自主设计”，结果就是卡死。后来改成给精确代码了。如果重做，我一开始就用精确代码策略，可以节约至少1小时。

3. **在开发环境里提前配置好代理。** 项目中遇到两次GitHub不可达的问题（`go get`超时），浪费了一些时间。应该在一开始就在开发环境里配好HTTP代理。

## 8. 对Superpowers这套方法论的批判

它的核心假设是：**人类做架构决策，AI做代码执行。** 这个假设在我的项目里基本成立——我主导了多工作区架构、极简封装哲学、CLI结构等决策。但问题在于，它低估了“人类需要把决策转译成AI能理解的精确语言”这个工作量。

另一个假设是：**任务可以拆成完全不耦合的小块。** 但真实项目里，像“pkg/types”这种共享类型定义，几乎所有的runner和executor都在用。写在task0里的话，后面所有task都需要依赖同一个类型定义。这种跨task的依赖没法通过“拆细”来完全解决——你需要依赖标注和全局一致性。

还有一个点：**技能缺少快进路径。** 当用户已经想清楚设计时，技能仍然按部就班问完所有问题。这浪费了经验丰富的用户的时间。

## 9. 对当前AI4SE工具和方法论的整体看法

做完这个项目后，我对“AI如何融入软件工程”的看法变得更务实了。我之前以为AI能“理解”一个需求然后自己写出来——实际上它并不能。它能做的是：你给我精确的规格，我用代码实现它，测试验证它。这个过程中，**人类的贡献不是在“写代码”这个层面，而是在“写规约”和“判断结果是否正确”这两个层面。**

所以这个项目对我来说最大的收获不是写了一个多仓库管理工具（虽然这本身也挺有成就感的），而是验证了一件事：**当AI能完成大部分编码工作时，工程师的核心能力变成了“能用精确的语言把需求说清楚”和“能判断AI写出来的东西到底对不对”。** 前者是架构师的技能，后者是评审官的技能。这两种技能，比单纯编码本身更值得训练。
